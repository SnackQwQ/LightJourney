# 旅行行程管理系统 — 需求分析文档

> 版本：v1.0 | 日期：2026-07-08
> 作者：AI驱动开发 | 状态：MVP阶段

---

## 一、项目背景

### 1.1 问题描述

旅行规划是高频刚需场景，但现有工具存在明显痛点：

- **传统方式（Excel/备忘录）**：手动填写效率低，行程调整麻烦，费用统计需人工计算
- **现有旅行App**：功能臃肿，捆绑酒店/机票销售，行程规划仍以手动为主，未充分利用AI能力
- **AI通用对话工具**：ChatGPT等可以生成旅行建议，但内容无法结构化保存、二次编辑和持续管理

### 1.2 产品定位

一款**轻量、克制、AI驱动**的行程管理工具。用户描述出行意图，AI生成结构化行程，一键保存管理。不做社区、不做电商、不做社交——只做"规划行程"这一件事。

### 1.3 核心价值

> 从"我想去成都玩3天，帮我规划一下"到拥有一份可编辑、可管理、可分享的完整行程表，只需要一次对话。

## 二、目标用户

### 2.1 用户画像

| 维度 | 描述 |
|------|------|
| 年龄段 | 20-40岁 |
| 旅行类型 | 自由行、周末短途游为主 |
| 行为特征 | 出发前会做攻略规划，有记录习惯 |
| 核心诉求 | 1) 快速生成合理行程 2) 集中管理多个目的地的计划 3) 方便分享给同行伙伴 |

### 2.2 用户场景举例

**场景一：临时起意的周末游**
> 小张周四晚上决定周末去杭州玩两天，打开系统输入"杭州、2天、偏好自然风光、预算1000"，AI生成的行程包括西湖骑行、龙井村茶园、九溪烟树等。小张把灵隐寺替换成了法喜寺，点保存。周五下班前在列表页看到行程，安心出发。

**场景二：多发地长期旅行规划**
> 小陈计划了一个西南环线游——成都2天、重庆2天、贵阳1天。她用系统分别规划了三段行程，在列表页按城市筛选查看，自动看到成都段预算860元、重庆段预算720元。发朋友圈前点"生成文案"，选了一条成都火锅行程，得到一段口语化的分享文字。

## 三、功能需求

### 3.1 MVP功能清单（第一轮）

#### F-01 用户注册

| 项 | 内容 |
|------|------|
| 描述 | 用户使用用户名和密码创建账号 |
| 输入 | 用户名（2-20字符，字母数字下划线）、密码（6-30字符） |
| 输出 | 注册成功，跳转登录页 |
| 规则 | 用户名唯一，重复时提示"用户名已被注册" |
| 异常 | 输入不合法时前端即时校验提示 |

#### F-02 用户登录

| 项 | 内容 |
|------|------|
| 描述 | 用户使用用户名和密码登录系统 |
| 输入 | 用户名、密码 |
| 输出 | 返回JWT access token，跳转行程列表页 |
| 规则 | token有效期24小时，存储在浏览器localStorage |
| 异常 | 用户名或密码错误时提示"用户名或密码错误" |

#### F-03 未登录拦截

| 项 | 内容 |
|------|------|
| 描述 | 未持有有效token的用户访问任何受保护页面时，自动跳转登录页 |
| 触发 | 前端路由守卫检测到无token或token已过期 |
| 行为 | 跳转 `/login`，登录后回到原目标页面 |

#### F-04 行程列表查看

| 项 | 内容 |
|------|------|
| 描述 | 登录后默认进入行程列表页，按日期分组展示当前用户所有行程 |
| 展示 | 行程卡片：城市标签、日期、时段、标题、预算金额 |
| 排序 | 按日期升序，同日期按时段升序 |
| 分组 | 不同日期之间用日期标题分隔 |
| 空状态 | 无行程时展示引导卡片"还没有行程？让AI帮你规划" |

#### F-05 行程筛选

| 项 | 内容 |
|------|------|
| 描述 | 支持按城市和日期范围筛选行程 |
| 筛选条件 | 城市（下拉，可选）、日期起止范围（日期选择器，可选） |
| 行为 | 两个条件可独立或组合使用，任一条件变更即时刷新列表 |
| 重置 | 清空筛选条件恢复显示全部行程 |

#### F-06 预算统计

| 项 | 内容 |
|------|------|
| 描述 | 自动计算并展示行程预算汇总 |
| 统计维度 | 1) 当前筛选结果总预算 2) 每日小计（在日期分组标题右侧显示） |
| 刷新 | 创建、编辑、删除行程或变更筛选条件时即时重新计算 |

#### F-07 新建行程

| 项 | 内容 |
|------|------|
| 描述 | 用户手动填写表单创建单条行程 |
| 必填字段 | 城市、日期、开始时间、结束时间、标题 |
| 选填字段 | 描述、预算 |
| 前端校验 | 1) 必填项非空 2) 结束时间必须晚于开始时间 |
| 后端校验 | 同时段冲突检测：同一用户同日期，新行程时段与已有行程重叠时禁止保存，提示"该时段与已有行程冲突" |

#### F-08 编辑行程

| 项 | 内容 |
|------|------|
| 描述 | 修改已有行程的任意字段 |
| 规则 | 与新建行程相同的校验规则 |
| 交互 | 点击行程卡片编辑按钮 → 弹出编辑表单（预填现有数据）→ 修改后保存 |

#### F-09 删除行程

| 项 | 内容 |
|------|------|
| 描述 | 删除单条行程 |
| 交互 | 点击行程卡片删除按钮 → 二次确认弹窗"确定删除该行程吗？"→ 确认后删除 |
| 规则 | 删除操作不可恢复 |

#### F-10 AI行程智能规划（Skill1）

| 项 | 内容 |
|------|------|
| 描述 | 用户输入出行意图，AI自动生成一套完整分时行程方案 |
| 输入参数 | 出行天数（1-30）、目的地城市、偏好（多选：美食/自然风光/人文历史）、预算上限（选填） |
| 行程粒度 | 每天2-3条行程（上午/下午/晚上），每条至少2小时 |
| 输出 | 结构化JSON数组，每条含日期、开始时间、结束时间、标题、描述（50-100字）、预算 |
| 交互流程 | ① 填写参数 → ② 预览AI生成的行程表格（可逐条展开编辑、删除、修改）→ ③ 确认保存 |

**预览编辑规则：**
- 每条行程可展开编辑面板，修改任意字段
- 可删除单条行程
- AI生成行程之间的时段冲突（内部冲突）以浅黄色背景提示，不阻断保存
- AI生成行程与已有行程的时段冲突（外部冲突）以红色标记，保存时硬拦截

**批量保存规则：**
- 无冲突的行程直接保存
- 有外部冲突的行程跳过
- 保存完成后弹窗明确告知结果："已保存X条，以下Y条因时段冲突未保存"

#### F-11 AI行程文案生成（Skill2）

| 项 | 内容 |
|------|------|
| 描述 | 选中单条行程，AI生成一段真实风格的朋友圈分享文案 |
| 触发 | 行程卡片操作区的"生成文案"按钮 |
| 输入 | 目标行程 + 当天同城市所有行程（提供上下文） |
| 风格 | 真实感分享型，口语化、带情绪，4-6句串联当天行程，适度使用1-2个emoji |
| 输出 | 一段可编辑的文本 |
| 交互 | 弹窗展示文案 → 用户可微调文字 → 一键复制 |

#### F-12 数据隔离

| 项 | 内容 |
|------|------|
| 描述 | 所有行程数据按用户隔离，用户A无法查看或操作用户B的行程 |
| 实现 | 所有行程接口从JWT中解析user_id，查询时强制带user_id条件 |

## 四、非功能需求

### 4.1 性能

| 指标 | 要求 |
|------|------|
| 页面首屏加载 | < 3秒（本地开发环境） |
| API响应时间（CRUD） | < 500ms |
| AI接口响应时间 | < 15秒（含DeepSeek生成时间） |
| AI请求超时 | 30秒后自动提示超时，允许重试 |

### 4.2 可用性

| 指标 | 要求 |
|------|------|
| 操作反馈 | 所有异步操作展示Loading状态 |
| 错误提示 | 统一使用全局消息提示（成功/失败），文案清晰 |
| 表单校验 | 前端即时校验 + 后端兜底校验 |
| 空状态 | 无数据时展示引导内容，避免空白页 |

### 4.3 安全性

| 项目 | 措施 |
|------|------|
| 密码存储 | bcrypt哈希，不存明文 |
| API密钥 | DeepSeek API Key仅存储在后端环境变量，前端不可见 |
| 鉴权 | 所有业务接口验证JWT，无token或过期拒绝访问 |
| 数据隔离 | 后端强制按user_id过滤，前端不可绕过 |
| SQL注入 | 使用ORM参数化查询 |
| 输入校验 | 前后端双重校验，拒绝非法输入 |

### 4.4 可维护性

| 项目 | 措施 |
|------|------|
| 代码结构 | 后端三层分离（路由/服务/数据访问），前端组件化 |
| Prompt管理 | AI Prompt模板集中在独立常量文件中，修改Prompt无需改动业务代码 |
| 数据库 | SQLite单文件，备份方便 |

## 七、约束与假设

### 7.1 技术约束

- 纯Web SPA，桌面端优先，移动端不做专门适配
- DeepSeek API作为唯一LLM服务，依赖其可用性
- SQLite作为唯一数据库，不引入外部数据库服务
- 本地开发运行，暂无生产部署需求

### 7.2 业务假设

- 用户有基本的旅行规划需求和使用Web应用的能力
- 用户对AI生成内容的准确性有合理预期（可能不完美，可二次编辑）
- 每个用户独立使用，不涉及行程共享
- 预算金额单位为人民币（元）
- 时段冲突仅在同一用户、同一日期内校验，跨天不校验

### 7.3 范围边界

- 不涉及真实地图数据和交通耗时计算（AI基于常识估算）
- 不涉及实时天气数据
- 不涉及支付、预订等交易功能
- 不分发/运营，仅作为个人工具使用

## 八、验收标准（MVP）

| 编号 | 验收项 | 通过标准 |
|------|--------|------|
| AC-01 | 注册登录 | 能注册新用户、登录获取token、token过期后拦截到登录页 |
| AC-02 | 行程CRUD | 能新建、编辑、删除行程，所有操作即时反映在列表页 |
| AC-03 | 时段冲突 | 同日期时段重叠时禁止保存，给出明确提示 |
| AC-04 | 筛选统计 | 按城市/日期筛选结果正确，预算统计自动更新 |
| AC-05 | AI规划 | 输入参数后AI返回可用的行程JSON，预览表格可编辑，确认后成功批量保存 |
| AC-06 | 文案生成 | 点击生成按钮返回口语化文案，可编辑和复制 |
| AC-07 | 数据隔离 | 用户A无法看到用户B的行程 |
| AC-08 | 空状态 | 新用户进入列表页展示引导内容而非空白 |
| AC-09 | 错误处理 | 网络异常、AI超时、输入非法时有明确提示，系统不崩溃 |
| AC-10 | AI生成不直写库 | AI返回内容必须先经过预览和用户确认才能写入数据库 |
